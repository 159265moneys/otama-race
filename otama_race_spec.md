# オタマレース（仮）― デモ仕様書 / 設計書 v1.0

> 体内を舞台に、精子オタマが卵子を目指す10人ぴょんぴょんレース。
> 本書は **コア検証用デモ** の仕様。製品全体ではなく「面白さの心臓が動くか」を最速で確かめるための設計。

---

## 0. このデモで検証したい1点

**「常時オート跳ね＋左右ブレ＋任意ジャンプ」のオタマ操作が、触っていて気持ちいいか / イライラか。**

ここがダメなら他の全要素（スキン・体内ギミック・マルチ）に意味がない。
だからデモは **まずソロ＋1コース** で手触りだけを見る。マルチ・スキン・ガチャは後回し。

## 1. ゲーム全体像（設計の確定事項）

### コンセプト（一行）
精子オタマが体内を駆け上がり、卵子に最初にたどり着いた1匹が受精＝勝ち。10人ぴょんぴょんパーティレース。

### 確定している設計判断
| 項目 | 決定 | 理由 |
|---|---|---|
| 人数 | **10人**（Steam P2P、重くならない上限） | カメレオン21人実証済の安全圏／10人は全員の事故が目で追える＝クリップになる |
| 進行構造 | **大枠固定**（膣→子宮頸管→子宮→卵管→卵子）／**中身ランダム** | 旅の進行感＋毎回違う鮮度。ローグライク型 |
| コース生成 | **確定層とランダム層をバームクーヘン状に交互配置** | 確定層＝見せ場・難易度アンカー、ランダム層＝飽き防止。自動生成の破綻を防ぐ |
| 部品 | ギミックを**アセット化してランダム呼び出し**。各部品はパラメータもランダム | 少数の部品で体感無限。個人開発で②枯渇を防ぐ |
| 勝利条件 | **脱落式＋最終1位受精の併用**（進行中CPで下位脱落→卵子前で残存数匹が1位を競う） | フォールガイズ型。near-missが構造に内蔵される |
| ビジュアル | 露骨にせず**意図はわかる範囲**（ポップ・かわいい寄り。リアル/キモい方向は不採用） | Steam審査クリア＋拡散しやすいトーン |
| プラットフォーム | Steam（PC） | P2P・売り切り・運用軽い |

### 判定器（この企画が満たしている設計フィルター8項目）
1. 一行で言える ◎
2. プレイヤーが無限にネタを生む ◎（体内題材×ランダム×妨害ドラマ）
3. 数秒で笑える/驚ける（バズる画）◎
4. 枯れた型の翻訳で軽実装 ◯（10人＋Steam任せ、同期はAIに委譲）
5. バズる画＝実際の体験（詐欺なし）◎
6. 運用が軽い（金溶けない・燃えない）◎
7. 配信者が仲間を誘う口実になる ◎
8. 少人数リアルタイムで成立（過疎耐性）◎

## 2. コア：オタマの操作仕様（デモの心臓）

### 2.1 移動モデル（2層構造）

**第1層：常時オート跳ね（ベース移動）**
- オタマは入力に関係なく、**常に一定リズムで小刻みに前方へ跳ね続ける**（おとなしくできない）
- 1回の跳ねごとに**左右にランダムな微ブレ**が乗る（まっすぐ入力してもまっすぐ進まない）
- プレイヤーの方向入力は「跳ねる向きをじわっと寄せる」程度＝**完全な制御はできない**
- → これが「思い通りにならない可愛さ／フォールガイズ的jank」の正体。**意図的な操作性の悪さ**

**第2層：任意ジャンプ（意思の操作）**
- オート跳ねとは**別に**、プレイヤーが任意発動する大ジャンプ
- 障害物越え・ショートカット・妨害回避に使う「ここぞ」の操作
- オート跳ね（小）＋任意ジャンプ（大）の2種類の高さが共存する

### 2.2 アクション一覧（デモ実装範囲）
| アクション | 入力 | 挙動 | デモ必須度 |
|---|---|---|---|
| オート跳ね | なし（常時） | 小刻みに前進＋左右微ブレ | **必須** |
| 方向入力 | スティック/キー | 跳ねる向きをじわっと寄せる | **必須** |
| 任意ジャンプ | ボタン1 | 大きく跳ぶ | **必須** |
| タメジャンプ | ボタン1長押し→離す | より高く/遠く跳ぶ | 必須 |
| ダッシュ | ボタン2 | 一時的に前進力UP | 推奨 |
| タメダッシュ | ボタン2長押し→離す | 強い加速 | 後回し可 |
| 回転攻撃 | ボタン3 | 周囲を弾く（対人妨害／障害排除） | 推奨（手触り確認用に1つは入れる） |
| 二段ジャンプ | ジャンプ中に再入力 | 空中でもう一段 | 後回し可 |

### 2.3 手触りチューニングの調整値（デモで触りながら詰める）
以下は**全てインスペクタ等で即変更できる変数**にしておくこと。手触りは数値の沼。
- オート跳ねの周期（跳ねる間隔）
- オート跳ねの高さ・前進距離
- **左右ブレの強さ**（最重要。弱いと退屈、強いとイライラ。気持ちいい一点を探す）
- 方向入力の効き（寄せる強さ）
- 任意ジャンプの高さ・初速
- タメ時間とタメ倍率
- 重力・落下速度

> **デモの成否判定：** 「ブレてまっすぐ進めないのに、なぜか もう一回やりたい」＝成功。
> 「ブレが理不尽でストレス」＝失敗。ここを数値で詰めるのがデモの主目的。

## 3. スキン仕様（数値設計・デモでは1〜2体でいい）

スキンは見た目＋**性能パラメータ**。7項目を各1〜5でスコアリングし、合計点でレア度を決める。

### 7つの能力値（各1〜5）
1. ジャンプ力
2. ダッシュ力
3. タメ時間（短いほど高評価＝高ポイント扱い）
4. 回転攻撃範囲
5. 自分の重さ
6. 自分の大きさ
7. ぴょんぴょん左右ブレ（小さいほど扱いやすい＝高ポイント扱い）

### レア度＝合計点の閾値
| レア度 | 合計点（例） |
|---|---|
| ノーマル | 〜20 |
| レア | 21〜24 |
| Sレア | 25〜29 |
| SSレア | 30〜 |

- **同レア内でも振り分けはバラバラ**（例：レア21点でも「ジャンプ全振り型」と「バランス型」が混在）
  → 同じレアでも個性が出る＝コレクション欲＆ビルド判断（ヴァンサバ的）
- デモでは**バランス型1体＋極端型1体**だけ作り、性能差が体感できるか確認

> ⚠️ デモではガチャ／コレクション画面は不要。性能差の体感だけ確認。

## 4. コース生成仕様（バームクーヘン構造）

### 4.1 レイヤー構造
```
[確定:スタート] 
  → [ランダム層:部品×2〜3抽選] 
    → [確定:見せ場①コンドーム乱闘] 
      → [ランダム層:部品×2〜3抽選] 
        → [確定:見せ場②8番赤水流] 
          → [ランダム層:部品×2〜3抽選] 
            → [確定:卵子前ラストスパート]
```
- **確定層**＝手設計の見せ場・難易度アンカー・記憶に残る瞬間。クリア可能性を保証するアンカーにもなる
- **ランダム層**＝部品プールから抽選。毎回違う

### 4.2 部品（ギミックアセット）の設計ルール
**接続規格を統一すること**（自動生成の破綻を防ぐ最重要点）：
- 各部品に**規格化された入口・出口**を持たせる（どう繋いでも必ず通行可能を保証）
- 「全員が通れる経路が必ず1本以上存在する」を部品単位で担保
- 部品ごとに**パラメータをランダム化**（敵の数・穴の位置/数・流速など）＝1部品が多数のバリエーションに化ける

### 4.3 難易度カーブ
- 進行（膣→卵管）ごとに**抽選する部品プールを変える**（序盤＝易しいプール、終盤＝難しいプール）
- → ランダムでも自然な難易度の起伏が出る

### 4.4 部品プール（初期に作る最低ライン：10〜15種）
体内題材は無限に増やせる。初期デモ用の例：
- 酸の床（触れると減速/脱落）
- 白血球（追尾して食べてくる動く障害物）
- 粘液（移動が重くなるゾーン）
- 子宮の収縮（道が周期的に狭まる＝タイミング）
- 水流（押し戻される／加速する流れ）
- **【確定見せ場】コンドームのボトルネック**（小さな穴に全員殺到→密集乱闘。回転攻撃が映える）
- **【確定見せ場】8番出口オマージュの赤い水流を泳ぐ**
- ※デモでは確定見せ場1つ＋ランダム部品3〜4種あれば検証可能

## 5. 勝利条件 / near-miss（脱落式＋最終1位）

### 5.1 基本ルール
- レースは複数チェックポイント（CP）で区切られる
- **各CPで規定順位より下のオタマが脱落**（フォールガイズ型）
- 最終区間（卵子前）で**残ったオタマが1位を競い、最初に卵子到達＝受精＝勝ち**

### 5.2 near-miss（要追加設計：デモで検証）
> ⚠️ **ここは今回確定させず、デモで手触りを見てから設計する項目。**
> 脱落式を入れた時点で near-miss の土台（あと一歩で脱落を免れる/食らう）は構造的に入る。
> その上で「惜しい→もう一回」を増幅する候補を、デモで試して選ぶ：
> - 候補A：僅差ゴール演出（あと一歩で抜かれた、を毎回ハッキリ見せる）
> - 候補B：チェックポイント復活（負けても途中から粘れる）
> - 候補C：妨害された瞬間のリプレイ/カルマ演出（あいつにやられた→次こそ）
>
> **near-missは中毒性の心臓。だが手触り次第なので、コア検証と同時にデモで詰める。**

## 6. ネットワーク（デモでは未実装）

- 製品では **Steam Networking API**（部屋管理・NAT越え・リレーを委譲）＋10人小部屋P2P
- レースは格ゲー/FPSより同期がシビアでない（多少のズレが許容される）。判定厳密化が要るのは「ゴール順位」「回転攻撃のヒット」のみ
- **デモではマルチを実装しない。** ソロ＋AIダミー（または無人）で**コアの手触りだけ**を見る
- マルチ同期の実装は手触りOK確定後、Claude Code等に委譲する重い工程

## 7. デモ開発の進め方（推奨順）

1. **オタマ1体・1直線コース**でオート跳ね＋左右ブレ＋任意ジャンプを実装。**手触りを数値で詰める**（最優先）
2. 回転攻撃を1つ足して、障害排除/妨害の手応えを見る
3. ランダム部品を3〜4種＋確定見せ場1つで、**バームクーヘン生成**を実装
4. CP脱落＋ゴールを入れて、**ソロで1レースが成立**するか
5. スキン2体で性能差の体感を確認
6. （ここまでOKなら）near-miss演出を試す → 手触り確定
7. 以降：マルチ同期・スキンガチャ・部品追加は別フェーズ

### 技術スタックの選択肢（未確定・次フェーズで判断）
- 3Dで体内を泳ぐ感じを出すなら：Godot or Unity
- コードのみで完結したい意向なら：Three.js（Webデモから入る手も）
